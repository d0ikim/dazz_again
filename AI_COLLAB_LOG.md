# AI 협업 로그

이 파일은 dazz_again을 진행하며 AI(Claude)와 함께 내린 프로세스/설계 결정과, AI가 자신있게 제시했지만 틀렸던 부분을 잡아낸 이력을 기록합니다.
이력서·면접 대비용 근거 자료(재료)로 쓰기 위한 목적이며, 회고가 아니라 "그 시점에 실제로 있었던 판단"을 그대로 남깁니다.

## 프로세스 결정 로그

### 2026-07-01 — AI 사용 방식: 훅 강제 대신 로그 방식으로 시작

**배경:** 타 개발자 이력서를 검토하며, AI 사용을 규율화한 시스템(계획 없는 구현을 hook으로 차단, ADR·PLAN·실패학습을 md로 기록, 컴파일 자가수정 3회 실패 시 사람에게 이관)이 핵심 재료임을 확인함.

**고민:** 지금 dazz_again의 우선순위는 MVP를 빠르게 완성해 실제 재즈 뮤지션들에게 배포하고, 실사용자 운영 경험까지 얻는 것. 파일마다 이해 확인을 강제하는 훅을 걸면 이 속도를 늦출 위험이 있음.

**결정:** 지금 단계에서는 강제(hook) 대신, 파일 완료·이해 확인 시점마다 이 로그에 기록만 남기는 방식으로 시작한다. 워크플로우에 마찰을 주지 않으면서 기록(재료)은 쌓이도록 함. 배포 후 실사용자 단계에서 AI가 만든 코드로 실질적 품질 문제(장애, 테스트 스킵 등)가 발생하면, 그때 hook 도입을 재검토한다.

**왜 이게 재료가 되는가:** "처음부터 규율을 강하게 걸었다"가 아니라 "속도와 품질의 트레이드오프를 상황에 맞게 판단해 단계적으로 도입했다"는 의사결정 자체가 면접에서 설명 가능한 지점.

### 2026-07-06 — 재즈바 크롤링 소스: 웹 스크래핑 대신 공식 API 병행으로 결정

**배경:** 서울의 재즈바를 최대한 수집해 venue 테이블에 넣는 크롤러가 필요. 후보는 ① Kakao Local API 키워드 검색, ② 네이버 지역검색 API, ③ 네이버지도/카카오맵 웹 스크래핑(Selenium).

**고민:** 스크래핑은 인스타 URL까지 얻을 수 있지만 JS 렌더링 의존이라 화면 구조가 바뀌면 바로 깨지고 약관 위반 소지가 있음. Kakao API는 안정적이고 좌표까지 한 번에 주지만 홈페이지/인스타 URL이 없음. 네이버 지역검색 API는 홈페이지 URL을 주지만 쿼리당 최대 5개라 단독 수집용으로는 부적합.

**결정:** Kakao 키워드 검색(지역별 키워드 × 3페이지)으로 이름·주소·좌표를 수집하고, 가게 이름별로 네이버 지역검색 API를 보조 호출해 홈페이지/인스타 URL만 보완하는 병행 구조로 결정. 중복은 수집 단계에서 카카오 장소 ID(set), INSERT 단계에서 이름 기준으로 이중 차단.

**같이 겪은 시행착오:** 첫 실행에서 "카테고리/이름에 재즈 포함" 필터가 너무 엄격해 13개만 수집됨(올댓재즈·클럽에반스 등 유명 재즈바가 칵테일바/호프로 분류돼 전부 탈락). Kakao 응답 카테고리를 실제로 찍어보고 나서 "검색어에 이미 재즈가 있으니 카카오의 연관성 판단을 신뢰하고, 카테고리 대분류(음식점/문화예술)로 노이즈만 차단"하는 방식으로 완화 → 127개 수집. 반대로 "서울 재즈카페" 키워드로 들어온 일반 커피숍 7개는 카테고리 재조회로 걸러 DELETE. **필터는 머리로 정하지 말고 실제 응답 데이터를 보고 정해야 한다**는 게 교훈.

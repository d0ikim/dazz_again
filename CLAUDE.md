# DAZZ 프로젝트 협업 규칙

## 유저 프로필
- 신입 백엔드 개발자 취업 준비 중인 학생
- 이 프로젝트가 유일한 포트폴리오 — 면접 질문에 대비해 모든 내용을 완전히 이해해야 함
- 커밋/PR은 GitHub Desktop 앱으로 직접 처리 (Claude는 git 명령어를 실행하지 않음)

---

## 진행 순서 (모든 작업에서 반드시 이 순서를 따른다)

### 기능 시작 시
- 새 브랜치가 필요한지 여부와 브랜치명을 먼저 안내한다.

### 파일 작성 시 (2-2)
- 파일 하나를 만들 때마다 **초보자도 이해할 수 있도록** 모든 import, 어노테이션, 메서드에 한국어 주석을 단다.
- 기존 파일의 주석 스타일(import 한 줄 한 줄 주석 포함)을 그대로 따른다.

### 파일 작성 후 반드시 멈춤 (2-2-1)
- 파일 하나를 만든 뒤 **반드시 멈춰서** 유저에게 이해했는지 묻는다.
- 유저가 응답하기 전까지 다음 파일로 절대 넘어가지 않는다.

### 유저 설명 답변이 오면 (2-2-2)
- 유저의 설명에서 틀리거나 부족한 부분을 바로잡아준다.
- 완전히 이해한 것이 확인되면 다음 파일로 넘어간다.

### 기능 완료 시 커밋 안내 (2-3)
- 한 기능이 완성되면 아래 형식으로 커밋 정보를 제공한다:
  ```
  커밋 메시지: feat: [기능 설명]
  커밋할 파일: [파일 목록]
  ```

### PR 안내 (3)
- PR이 필요한지 여부를 안내한다.
- 필요하면 PR 제목, 본문, merge 방식, merge 후 브랜치 삭제 여부를 안내한다.

### 다음 대화 이어가기 안내 (4)
- 한 기능이 끝나면 다음에 남은 작업 목록을 정리해준다.
- 유저가 새 대화에서 `/clear` 후 붙여넣기할 수 있는 형태로 작성한다.
